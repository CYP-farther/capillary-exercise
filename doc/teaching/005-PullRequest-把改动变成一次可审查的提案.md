# Pull Request：把每次改动变成一次可审查的提案

> 教学材料 · 流程与质量篇（承接 004 代码规范，引出 006 CI / 007 Code Review）

前面我们把代码怎么写（004 规范）讲清楚了。但代码写好之后，**怎么把它"交进"主干**？

直接在 `main` 上改、改完就完事吗？这一篇讲一个绕不开的工程动作：**Pull Request（PR，拉取请求）**——它是 CI 自动验证和 Code Review 人工审查共同的"载体"。后面两篇（CI、Code Review）讲的事，全都发生在 PR 上，所以先把 PR 讲明白。

这一篇回答三个问题：**PR 是什么、为什么要用 PR、怎么用 PR。**

---

## 一、PR 是什么

一句话：

> **PR 是"我想把这条分支的改动合并进主干，请过一遍"的一次正式提案。**

拆开看，它包含几样东西：

- **一条分支的改动**：你在 `feature/issue-7-pickup-service` 上写的所有提交
- **一个目标**：请求合并进 `main`
- **一段说明**：你做了什么、为什么这么做、怎么验证的
- **一个讨论与检查的场所**：CI 在这里自动跑，审查意见在这里留，所有人能看到改了哪些行

它不是"把文件传上去"，而是**把一组改动打包成一个可以被讨论、被检查、被批准或打回的对象**。

### 和"直接提交到 main"的对比

| | 直接改 main | 走 PR |
|---|------------|-------|
| 改动可见性 | 改完才发现，散落在历史里 | 合并前就能整体看 diff |
| 自动检查 | 没有触发点 | PR 触发 CI（编译/测试） |
| 人工把关 | 没有关卡 | 合并前可审查 |
| 可追溯 | 一堆零散 commit | 一个 PR 关联一个 Issue，前因后果齐全 |
| 出错回退 | 翻历史、挑 commit | 整个 PR 一起回退 |

**直接改 main，等于"边写边发布"，没有任何缓冲。PR 是主干和你的改动之间的那道缓冲门。**

---

## 二、为什么要用 PR

### 1. 它给"自动检查"和"人工审查"一个挂载点

这是最实际的理由。CI 凭什么知道"现在该跑测试了"？Code Review 在哪里进行？——**答案都是 PR**。

```
没有 PR：改动直接进 main，CI 没有触发时机，审查没有发生的场所
有  PR：提 PR → 自动触发 CI → CI 绿了 → 在 PR 上审查 → 通过才合并
```

PR 是那个"提交点"。没有它，后面两篇讲的 CI 和 Code Review 都无处安放。

### 2. 它强迫你把改动"成块"地表达清楚

提 PR 要写一段描述：**做了什么、为什么这么做**。

这件事看着是负担，其实是**第二次思考**。如果你发现自己写不清楚这段描述——"我到底改了啥，为什么这么改"——那往往说明改动本身就没想清楚（这一点 007 Code Review 还会再讲）。

> PR 描述是一面镜子，照出你脑子里的含糊。

### 3. 它让每次改动可追溯、可回退

一个 PR 对应一个 Issue（`Closes #7`），合并后你能顺着 PR 回看：这次改动解决了哪个问题、改了哪些文件、当时为什么这么决定。

万一出问题，整个 PR 可以作为一个整体回退，而不用在一堆零散提交里大海捞针。

### 4. 单人项目也值得——"提案 → 检查 → 合并"是一种纪律

"就我一个人，自己给自己提 PR，不是脱裤子放屁？"

不是。和 006/007 反复讲的一样：**PR 制造了一个"停顿点"**。在合并前，它逼你停下来——让 CI 跑一遍、让你以"审查者"的角度再看一眼整块改动。

没有这个停顿，单人开发很容易变成"写到哪算哪、随手就进主干"，问题悄悄沉淀。PR 把"提交"这个动作仪式化，**给随性的开发装上一道闸**。

## 三、怎么用 PR

落到本项目的真实流程。每个 Issue 都走同一套（见 `CLAUDE.md` 的迭代开发流程）：

```
1. 建分支    git checkout -b feature/issue-7-pickup-service
2. 编码      遵守 004 代码规范
3. 本地测试   dotnet test（先在本地绿一遍）
4. 提交推送   git commit（关联 Issue）→ git push
5. 提 PR     gh pr create，写清"做了什么、为什么"，正文带 Closes #7
                  ↓
6. CI 验证   PR 自动触发 GitHub Actions：编译 + 跑全部测试（见 006）
                  ↓
7. Code Review  在 PR 上审设计与质量（见 007）
                  ↓
8. 合并      CI 绿 + 审查通过 → 合并进 main → Issue 自动关闭
```

### 关键习惯

- **一个 PR 只做一件事**：对应一个 Issue。别把"实现领料"和"顺手重构日志"塞进同一个 PR——审查者会看不清，回退也牵连。
- **分支名能说明意图**：`feature/issue-7-pickup-service` 一看就知道在干嘛。
- **PR 描述写清三件事**：做了什么、为什么这么做、怎么验证的。
- **关联 Issue**：正文写 `Closes #7`，合并时 Issue 自动关闭，改动和需求挂上钩。
- **CI 没绿不合并**：这是底线（006 细讲）。

### 本项目的真实痕迹

本项目领料流程的 8 个 Issue，**每一个都走了一次 PR**（#14、#17、#18、#20、#21、#22、#23、#24），一个 PR 关一个 Issue。你现在去仓库翻这些 PR，还能看到当时每次改了什么、为什么——这就是 PR 留下的可追溯记录。

> 归档的 Issue 见 `doc/issue-design/`，开发全过程的复盘见 `teaching/030`。

## 四、PR 在整个流程里的位置

PR 不是孤立的一步，它是个**容器**——CI 和 Code Review 都装在里面：

```
            ┌─────────────── 一个 PR ───────────────┐
建分支 编码 →│ 提 PR → CI 自动验证 → Code Review → 合并 │→ main
            └────────────────────────────────────────┘
                         ↑               ↑
                    006 讲这里        007 讲这里
```

所以顺序是：**先有 PR（005，这一篇）这个容器，CI（006）才有地方自动跑，Code Review（007）才有地方进行。** 这也是为什么这三篇按 PR → CI → Review 的顺序讲。

## 五、最后：聊聊"门槛"

和这个系列一贯的视角一样，说一句门槛。

过去，**规范地走 PR 流程，对单人或小项目是有成本的**：要建分支、写描述、配置自动检查……很多人嫌麻烦，干脆直接 push 到 main，PR 流程只在大团队里才用得起。**这情有可原**：当年的工具链确实重。

**但今天，门槛降得很低了。**
- `gh pr create` 一条命令就建好 PR
- GitHub Actions 几行配置就让 PR 自动跑 CI（006 会讲）
- AI 能帮你起草 PR 描述、甚至在 PR 上当"第二双眼睛"审一遍（007 会讲）

> 过去：PR 流程成本高 → 单人/小项目跳过，直接改 main，情有可原
> 现在：一条命令建 PR、几行配置接 CI → 单人项目也能用上规范的"提案 → 检查 → 合并"

哪怕你一个人做项目，也值得给每次改动走一次 PR——它花不了几分钟，却给你换来了自动检查的挂载点、合并前的停顿、和一份可追溯的记录。

---

## 一句话总结

> **PR 是把一条分支的改动打包成"一次可审查的提案"——它是 CI 自动验证和 Code Review 人工审查共同的载体。**
> **一个 PR 对应一个 Issue，写清"做了什么、为什么"，CI 绿了、审查过了再合并。哪怕单人项目，这道"提案 → 检查 → 合并"的闸门也值得装上。**
