# 软件演示指南 · 现场跑通领料流程

> 教学材料 · 演示操作篇 · 配套实操

---

## 这篇是干什么的

前面 000–020 讲的是「为什么这样做」。这一篇不讲理念，只讲一件事：

**怎么把这个软件当场跑起来，让观众看到领料流程的成功与各种失败分支。**

照着下面的步骤走，10 分钟内可以完整演示一遍。建议演示前自己先按一遍，把数据库重置好（见下文「演示前重置」）。

---

## 一、它是个什么程序

- **C# .NET 9 WinForms 桌面程序**，单窗口，标题「劈刀领料」。
- **无需真实硬件**：PLC、扫码器、MES 全部是进程内 Fake（假实现），所以在任何一台装了 .NET 9 SDK 的 Windows 机器上都能直接跑。
- **数据库是本地 SQLite 单文件**（`CapillaryData.db`），首次启动自动建表并预置演示数据。
- 演示的核心是**领料流程**：输入工单号 + 机台号 → 查 MES 要类型 → FIFO 找库存 → PLC 取料 → 扫码验证 → MES 上报 → 出料。

---

## 二、环境准备（演示机器上确认一次）

1. 装了 **.NET 9 SDK**。命令行验证：

   ```
   dotnet --version
   ```

   能输出 `9.0.xxx` 即可。

2. 是 **Windows** 系统（WinForms 只能在 Windows 上跑界面）。

3. 拿到项目代码，进入项目根目录 `capillary-exercise`。

> 演示机如果没装 SDK 又不方便装，可在有 SDK 的机器上 `dotnet publish` 出一个自包含 exe 拷过去。现场培训一般直接用 SDK 跑，最省事。

## 三、启动程序

在项目根目录执行：

```
dotnet run --project src/CapillaryExercise/CapillaryExercise.csproj
```

第一次会编译几秒，然后弹出「劈刀领料」窗口：

```
┌─ 劈刀领料 ──────────────────────────────┐
│  演示案例 [✅ 成功领料（WO001） ▼] ┌──────────┐ │
│                                  │重置演示数据│ │
│           预期：完整流程跑通…      └──────────┘ │
│  工单号  [WO001_______]  ┌────────┐         │
│  机台号  [M1__________]  │开始领料│         │
│                          └────────┘         │
│  ┌─ 领料进度 ───────────────────────────┐  │
│  │   (这里逐行显示每一步)               │  │
│  └──────────────────────────────────────┘  │
│  [          结果栏（成功绿/失败红）        ] │
└──────────────────────────────────────────────┘
```

操作很简单：在顶部**演示案例**下拉里选一个，工单号/机台号会自动填好、下方还给出**预期结果**提示，然后点**开始领料**，看进度栏逐步刷新、结果栏出结论。

> **不用记工单号了**。早先的版本要手动输入工单号（且大小写敏感，输成 `wo001` 会直接 MES 查询失败），现在改成下拉选案例自动填参，省掉了这个坑。你仍然可以手动改输入框里的值做自由测试——比如把工单号清空验证「必填」校验，或随便敲个没配过的号看 MES 失败。
>
> **重置演示数据**按钮：一键把库存恢复到初始演示状态（详见第五节）。

## 四、四个演示场景（下拉直接选）

程序首次启动预置了一批演示数据，配好了四个内置案例，分别命中**成功**和三种**失败**分支。下拉框里**已按「先成功后失败」排好序**，从上到下依次演示即可。

| 顺序 | 演示案例（下拉项） | 自动填入 | 预期结果 | 演示的点 |
|------|------------------|----------|----------|----------|
| 1 | ✅ 成功领料（WO001） | WO001 / M1 | ✅ 领料成功 | 完整正向流程跑通 |
| 2 | ❌ 库存不足（WO003） | WO003 / M1 | ❌ 库存不足: CAP-C | 没货时的拦截 |
| 3 | ❌ 读码失败·仓位锁定（WO002） | WO002 / M1 | ❌ 读码失败，仓位已锁定 | 取出来但扫码不符 → 放回 + 锁仓 |
| 4 | ❌ 工单无效（WO999） | WO999 / M1 | ❌ MES查询失败或工单无效 | 工单本身无效 |

### 场景 1：成功领料（WO001）

选下拉里的**「✅ 成功领料（WO001）」**（工单号/机台号自动填 `WO001` / `M1`），点开始。进度栏逐行出现：

```
✓ MES: 查询劈刀类型...
✓ 查找库存: CAP-A...
✓ PLC: 从 A0510 取料...
✓ 扫码器: 读码验证...
✓ MES: 上报领料信息...
✓ PLC: 出料...
```

结果栏**绿色**：`领料成功: BC-A-001`。

讲解点：这一条龙就是设计文档里的领料主流程。WO001 在 MES 里对应类型 CAP-A，系统按 **FIFO** 取了最早入库的那支（BC-A-001），扫码器读出来的码和数据库记录一致，验证通过，上报、出料、更新库存为「已领出」。

### 场景 2：库存不足（WO003）

选**「❌ 库存不足（WO003）」**，点开始。WO003 对应类型 CAP-C，但库里**没有预置任何 CAP-C**。进度走到查库存就停：

```
✓ MES: 查询劈刀类型...
✗ 查找库存: CAP-C...
```

结果栏**红色**：`库存不足: CAP-C`。

讲解点：MES 能查到类型，但仓里没货，流程在第二步就被拦下，**不会去动 PLC**。

### 场景 3：读码失败、仓位锁定（WO002）

选**「❌ 读码失败·仓位锁定（WO002）」**，点开始。WO002 对应 CAP-B，库里有一支 BC-B-001。但**扫码器在演示里只认 CAP-A 的那个码**，取出 CAP-B 后扫出来的码对不上：

```
✓ MES: 查询劈刀类型...
✓ 查找库存: CAP-B...
✓ PLC: 从 B0307 取料...
✗ 扫码器: 读码验证...
  读码失败，放回原位并锁定...
```

结果栏**红色**：`读码失败，仓位已锁定`。

讲解点：这是个**异常处理**的好例子。劈刀已经被取出来了，但扫码验证不通过——系统的策略是**放回原仓位 + 把该仓位锁定**（状态置为「锁定」，不再参与后续 FIFO），并记日志。它不是简单报错了事，而是保证了物料和数据的一致。

### 场景 4：工单无效 / MES 查询失败

选**「❌ 工单无效（WO999）」**，点开始。MES 里查不到 WO999 这个工单：

```
✗ MES: 查询劈刀类型...
```

结果栏**红色**：`MES查询失败或工单无效`。

讲解点：流程在**第一步**就被拦住。工单号是整个流程的入口，进不来就什么都不做。（WO999 只是个没在 MES 里配过的号，随便换成别的没配过的也一样。）

## 五、重复演示与重置（重要，容易翻车的点）

**数据库是持久化的**——库存状态会跨次保留，有两个场景跑一次后状态就变了：

- 场景 1（WO001）**成功一次后**，那支 BC-A-001 标记为「已领出」。再选 WO001，FIFO 会取下一支 BC-A-002，但扫码器仍只认 BC-A-001 的码 → **第二次会变成「读码失败」**，不再是成功。
- 场景 3（WO002）**跑过一次后**，那支 CAP-B 被锁定。再选 WO002 → 库里没有可用 CAP-B → **变成「库存不足」**。

### 最简单的办法：点「重置演示数据」按钮

界面右上角的**「重置演示数据」**按钮就是为这个准备的：点一下，程序会清空库存与日志、重新预置那批演示劈刀，**四个场景立即全部复位**，结果栏会提示「演示数据已重置」。**不用关程序、不用删文件、不用重启。**

> 演示建议：每演示完一轮（尤其演过「成功」或「读码失败」之后），点一下「重置演示数据」，再开始下一轮，结果就永远和文档一致。

### 备选：删库文件（一般用不上）

如果你想连数据库文件本身都清掉（比如换演示机、或怀疑库被改乱了），可以关掉程序后删文件，下次启动会重新建库预置：

```
# 在项目根目录（Git Bash / PowerShell 下 rm 亦可）
del src\CapillaryExercise\bin\Debug\net9.0-windows\CapillaryData.db
```

> 有了重置按钮后，日常演示基本不需要这一步。

## 六、预置数据速查（讲解时心里有数）

首次启动 / 重置后，库里有这些在库劈刀：

| 条码 | 类型 | 仓位 | 入库时间 | 备注 |
|------|------|------|----------|------|
| BC-A-001 | CAP-A | A(05,10) | 2026-01-01 | FIFO 最早，WO001 命中它，扫码器认这个码 → 成功 |
| BC-A-002 | CAP-A | A(05,11) | 2026-01-02 | CAP-A 第二支，BC-A-001 被领走后才轮到它 |
| BC-B-001 | CAP-B | B(03,07) | 2026-01-01 | WO002 命中它，但扫码器不认它的码 → 读码失败 |
| （无） | CAP-C | — | — | 故意不预置 → WO003 演示「库存不足」 |

工单 → 类型映射（在 MES 假实现里配死）：

| 工单 | 类型 | 结局 |
|------|------|------|
| WO001 | CAP-A | 成功 |
| WO002 | CAP-B | 读码失败、锁仓 |
| WO003 | CAP-C | 库存不足 |
| 其它 | （查不到） | MES 查询失败 |

> 这些预置逻辑写在 `src/CapillaryExercise/Demo/DemoDataSeeder.cs`（库存数据）和 `Program.cs`（`WithType(...)` 工单映射 + `BuildDemoScenarios()` 案例列表）里。想加场景就改这几处。

## 七、可选：顺带演示「无需真实硬件」这件事

如果观众里有工程背景，可以打开 `Program.cs` 给他们看这几行：

```csharp
IPlcController plc = new FakePlcController();
IScanner scanner = new FakeScanner(DemoDataSeeder.DemoBarcode);
IMesService mes = new FakeMesService()...
```

讲解点：业务逻辑（`PickupService`）**只依赖接口**（`IPlcController` / `IScanner` / `IMesService`），不关心背后是真硬件还是假实现。今天演示用的是进程内 Fake，**换到产线只要把这三行换成真正的 FINS / 串口 / HTTP 实现，业务代码一行不用改**。这正是「面向接口编程 + 依赖注入」在演示里能直接看到的好处，呼应 004（代码规范）和设计文档第六节。

## 九、出问题时的快速排查

| 现象 | 原因 / 处理 |
|------|------------|
| `dotnet run` 报找不到命令 | 没装 .NET 9 SDK，或没配进 PATH |
| 弹窗一闪而过或报错退出 | 多半是上次的程序没关，数据库文件被占用；关掉所有实例再试 |
| WO001 点了变成「读码失败」 | 之前成功领过一次，BC-A-001 已被领走。点**「重置演示数据」**即可复位（见第五节） |
| WO002 点了变成「库存不足」 | 之前跑过一次把 CAP-B 锁了。点**「重置演示数据」**复位 |
| 进度栏没反应、按钮一直灰着 | 正常情况执行很快就恢复；若卡住，关程序重开 |
| 界面是英文/乱码 | 标题应是「劈刀领料」，乱码一般是终端编码问题，不影响程序本身 |

## 十、演示流程串词（30 秒版）

> 「这是一个劈刀自动领料系统。我在演示案例里选‘成功领料’，工单号机台号自动填好，点开始——大家看进度栏，它在跟 MES 要劈刀类型、按先进先出找库存、控制 PLC 取料、扫码验证、上报、出料，一气呵成，成功。
>
> 现在我选‘读码失败’这个案例（工单 WO002），它对应的劈刀虽然有货，但取出来扫码对不上——看，系统没有硬来，而是把它放回原位、锁定这个仓位，保证物料和数据一致。
>
> 再选‘库存不足’（WO003），这种劈刀没库存，流程在找库存这一步就停了，不会去碰硬件。
>
> （演示完点一下‘重置演示数据’，库存就回到初始状态，可以再来一轮。）
>
> 关键是：今天没接任何真实硬件，PLC、扫码器、MES 全是假实现。但因为代码是面向接口写的，换到产线只改三行就能接真设备，业务逻辑一行不动。」
