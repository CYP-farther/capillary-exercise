# 软件演示指南 · 现场跑通领料流程

> 教学材料 · 演示操作篇 · 配套实操

---

## 这篇是干什么的

前面 000–020 讲的是「为什么这样做」。这一篇不讲理念，只讲一件事：

**怎么把这个软件当场跑起来，让观众看到领料流程的成功与各种失败分支。**

照着下面的步骤走，10 分钟内可以完整演示一遍。建议演示前自己先按一遍，把数据库重置好（见下文「演示前重置」）。

---

## 一、它是个什么程序

- **C# .NET 9 WinForms 桌面程序**，单窗口，标题「劈刀领料」。
- **无需真实硬件**：PLC、扫码器、MES 全部是进程内 Fake（假实现），所以在任何一台装了 .NET 9 SDK 的 Windows 机器上都能直接跑。
- **数据库是本地 SQLite 单文件**（`CapillaryData.db`），首次启动自动建表并预置演示数据。
- 演示的核心是**领料流程**：输入工单号 + 机台号 → 查 MES 要类型 → FIFO 找库存 → PLC 取料 → 扫码验证 → MES 上报 → 出料。

---

## 二、环境准备（演示机器上确认一次）

1. 装了 **.NET 9 SDK**。命令行验证：

   ```
   dotnet --version
   ```

   能输出 `9.0.xxx` 即可。

2. 是 **Windows** 系统（WinForms 只能在 Windows 上跑界面）。

3. 拿到项目代码，进入项目根目录 `capillary-exercise`。

> 演示机如果没装 SDK 又不方便装，可在有 SDK 的机器上 `dotnet publish` 出一个自包含 exe 拷过去。现场培训一般直接用 SDK 跑，最省事。

## 三、启动程序

在项目根目录执行：

```
dotnet run --project src/CapillaryExercise/CapillaryExercise.csproj
```

第一次会编译几秒，然后弹出「劈刀领料」窗口：

```
┌─ 劈刀领料 ──────────────────────────┐
│  工单号  [____________]  ┌────────┐  │
│  机台号  [____________]  │开始领料│  │
│                          └────────┘  │
│  ┌─ 领料进度 ─────────────────────┐  │
│  │                                │  │
│  │   (这里逐行显示每一步)         │  │
│  │                                │  │
│  └────────────────────────────────┘  │
│  [        结果栏（成功绿/失败红）   ] │
└──────────────────────────────────────┘
```

操作很简单：填**工单号**和**机台号**，点**开始领料**，看进度栏逐步刷新、结果栏出结论。

> 机台号在演示里**填任意非空值都行**（比如 `M1`），它不影响分支走向，但不能留空——留空会直接提示「请输入工单号和机台号」。真正决定演示走向的是**工单号**。

## 四、四个演示场景（按工单号触发）

程序首次启动预置了一批演示数据，配好了四条工单，分别命中**成功**和三种**失败**分支。**按下面顺序演示效果最好**：先成功，再逐个失败。

| 顺序 | 工单号 | 机台号 | 预期结果 | 演示的点 |
|------|--------|--------|----------|----------|
| 1 | `WO001` | 任意（如 `M1`） | ✅ 领料成功 | 完整正向流程跑通 |
| 2 | `WO003` | 任意 | ❌ 库存不足: CAP-C | 没货时的拦截 |
| 3 | `WO002` | 任意 | ❌ 读码失败，仓位已锁定 | 取出来但扫码不符 → 放回 + 锁仓 |
| 4 | `WO999`（任意没配过的号） | 任意 | ❌ MES查询失败或工单无效 | 工单本身无效 |

### 场景 1：成功领料（WO001）

输入 `WO001` / `M1`，点开始。进度栏逐行出现：

```
✓ MES: 查询劈刀类型...
✓ 查找库存: CAP-A...
✓ PLC: 从 A0510 取料...
✓ 扫码器: 读码验证...
✓ MES: 上报领料信息...
✓ PLC: 出料...
```

结果栏**绿色**：`领料成功: BC-A-001`。

讲解点：这一条龙就是设计文档里的领料主流程。WO001 在 MES 里对应类型 CAP-A，系统按 **FIFO** 取了最早入库的那支（BC-A-001），扫码器读出来的码和数据库记录一致，验证通过，上报、出料、更新库存为「已领出」。

### 场景 2：库存不足（WO003）

输入 `WO003` / `M1`。WO003 对应类型 CAP-C，但库里**没有预置任何 CAP-C**。进度走到查库存就停：

```
✓ MES: 查询劈刀类型...
✗ 查找库存: CAP-C...
```

结果栏**红色**：`库存不足: CAP-C`。

讲解点：MES 能查到类型，但仓里没货，流程在第二步就被拦下，**不会去动 PLC**。

### 场景 3：读码失败、仓位锁定（WO002）

输入 `WO002` / `M1`。WO002 对应 CAP-B，库里有一支 BC-B-001。但**扫码器在演示里只认 CAP-A 的那个码**，取出 CAP-B 后扫出来的码对不上：

```
✓ MES: 查询劈刀类型...
✓ 查找库存: CAP-B...
✓ PLC: 从 B0307 取料...
✗ 扫码器: 读码验证...
  读码失败，放回原位并锁定...
```

结果栏**红色**：`读码失败，仓位已锁定`。

讲解点：这是个**异常处理**的好例子。劈刀已经被取出来了，但扫码验证不通过——系统的策略是**放回原仓位 + 把该仓位锁定**（状态置为「锁定」，不再参与后续 FIFO），并记日志。它不是简单报错了事，而是保证了物料和数据的一致。

### 场景 4：工单无效 / MES 查询失败（任意没配过的号）

输入 `WO999` / `M1`（或任何不是 WO001/2/3 的号）。MES 里查不到这个工单：

```
✗ MES: 查询劈刀类型...
```

结果栏**红色**：`MES查询失败或工单无效`。

讲解点：流程在**第一步**就被拦住。工单号是整个流程的入口，进不来就什么都不做。

## 五、演示前重置（重要，容易翻车的点）

**数据库是持久化的**——程序退出后 `CapillaryData.db` 还在，里面的库存状态会保留上一次演示的结果。这会导致**重复演示时结果变样**：

- 场景 1（WO001）**成功一次后**，那支 BC-A-001 就标记为「已领出」了。再点一次 WO001，FIFO 会取下一支 BC-A-002，但扫码器仍只认 BC-A-001 的码 → **第二次会变成「读码失败」**，不再是成功。
- 场景 3（WO002）**跑过一次后**，那支 CAP-B 被锁定。再点 WO002 → 库里没有可用 CAP-B 了 → **变成「库存不足」**。

所以：**每次想从干净状态完整演示一遍，先删掉数据库文件再启动。**

数据库文件在程序输出目录：

```
src/CapillaryExercise/bin/Debug/net9.0-windows/CapillaryData.db
```

重置方法（程序先关掉，再删文件）：

```
# 在项目根目录
del src\CapillaryExercise\bin\Debug\net9.0-windows\CapillaryData.db
```

> Git Bash / PowerShell 下用 `rm` 也行。删掉后下次 `dotnet run` 启动会重新建表 + 预置那批演示数据，四个场景全部复位。

**演示建议**：如果你要反复演示「成功」这个场景，最稳妥的是**每演示一轮就关程序、删库、重启**。或者把「成功」放在每轮第一个演示、且一轮只演一次。

## 六、预置数据速查（讲解时心里有数）

首次启动 / 重置后，库里有这些在库劈刀：

| 条码 | 类型 | 仓位 | 入库时间 | 备注 |
|------|------|------|----------|------|
| BC-A-001 | CAP-A | A(05,10) | 2026-01-01 | FIFO 最早，WO001 命中它，扫码器认这个码 → 成功 |
| BC-A-002 | CAP-A | A(05,11) | 2026-01-02 | CAP-A 第二支，BC-A-001 被领走后才轮到它 |
| BC-B-001 | CAP-B | B(03,07) | 2026-01-01 | WO002 命中它，但扫码器不认它的码 → 读码失败 |
| （无） | CAP-C | — | — | 故意不预置 → WO003 演示「库存不足」 |

工单 → 类型映射（在 MES 假实现里配死）：

| 工单 | 类型 | 结局 |
|------|------|------|
| WO001 | CAP-A | 成功 |
| WO002 | CAP-B | 读码失败、锁仓 |
| WO003 | CAP-C | 库存不足 |
| 其它 | （查不到） | MES 查询失败 |

> 这些预置逻辑写在 `src/CapillaryExercise/Program.cs` 里（`SeedDemoDataIfEmpty` 和那几行 `WithType(...)`）。想加场景就改这里。

## 七、可选：顺带演示「无需真实硬件」这件事

如果观众里有工程背景，可以打开 `Program.cs` 给他们看这几行：

```csharp
IPlcController plc = new FakePlcController();
IScanner scanner = new FakeScanner(demoBarcode);
IMesService mes = new FakeMesService()...
```

讲解点：业务逻辑（`PickupService`）**只依赖接口**（`IPlcController` / `IScanner` / `IMesService`），不关心背后是真硬件还是假实现。今天演示用的是进程内 Fake，**换到产线只要把这三行换成真正的 FINS / 串口 / HTTP 实现，业务代码一行不用改**。这正是「面向接口编程 + 依赖注入」在演示里能直接看到的好处，呼应 004（代码规范）和设计文档第六节。

## 九、出问题时的快速排查

| 现象 | 原因 / 处理 |
|------|------------|
| `dotnet run` 报找不到命令 | 没装 .NET 9 SDK，或没配进 PATH |
| 弹窗一闪而过或报错退出 | 多半是上次的程序没关，数据库文件被占用；关掉所有实例再试 |
| WO001 点了变成「读码失败」 | 之前成功领过一次，BC-A-001 已被领走。**删库重启**（见第五节） |
| WO002 点了变成「库存不足」 | 之前跑过一次把 CAP-B 锁了。**删库重启** |
| 进度栏没反应、按钮一直灰着 | 正常情况执行很快就恢复；若卡住，关程序重开 |
| 界面是英文/乱码 | 标题应是「劈刀领料」，乱码一般是终端编码问题，不影响程序本身 |

## 十、演示流程串词（30 秒版）

> 「这是一个劈刀自动领料系统。我输入工单号 WO001、机台号 M1，点开始——大家看进度栏，它在跟 MES 要劈刀类型、按先进先出找库存、控制 PLC 取料、扫码验证、上报、出料，一气呵成，成功。
>
> 现在我换个工单 WO002，它对应的劈刀虽然有货，但取出来扫码对不上——看，系统没有硬来，而是把它放回原位、锁定这个仓位，保证物料和数据一致。
>
> 再换 WO003，这种劈刀没库存，流程在找库存这一步就停了，不会去碰硬件。
>
> 关键是：今天没接任何真实硬件，PLC、扫码器、MES 全是假实现。但因为代码是面向接口写的，换到产线只改三行就能接真设备，业务逻辑一行不动。」
