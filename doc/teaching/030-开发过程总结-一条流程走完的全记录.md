# 开发过程总结 · 一条流程走完的全记录

> 教学材料 · 收尾篇 · 过程复盘

---

## 这篇是干什么的

000–020 讲的是「为什么要这样做」，040 讲的是「怎么把成品跑起来演示」。

这一篇夹在中间，回答另一个问题：**我们到底走了一条什么样的路，才从一句话需求走到一个能跑、有测试、有 CI 的软件？**

它是一次**复盘**——把整个开发过程拉成一条时间线，让你看清每一步的产出物、它们之间的依赖，以及中途我们怎么发现问题、修正方案。读完它，前面所有零散的「为什么」会串成一个整体。

---

## 一、一句话概括

> 从「做一个劈刀领料系统」这句话出发，先后产出**需求 → 设计 → Issue 拆解 → 测试计划 → 代码规范/CI** 五份底座文档，再按 Issue 逐个迭代开发，最终交付一个**无需真实硬件即可运行、36 个测试全绿、每次提交自动验证**的 .NET 9 WinForms 程序。

整个过程**没有一行业务代码是在想清楚之前写的**。

---

## 二、完整时间线（按真实 commit 顺序）

### 阶段 0：开张（环境与仓库）

- 装 Git / GitHub CLI，建仓库 `capillary-exercise`，搭目录骨架（`doc/` `src/` `test/` `.github/`）。
- 立下**文档编号规则**（001-需求、002-设计……）和 **Git 习惯**（每阶段独立提交、提交信息说清做了什么和为什么）。

> 关键动作：还没写任何代码，先把"东西放哪、怎么提交"定下来。

### 阶段 1：想清楚要做什么、怎么搭（三份设计文档）

| 产出 | 文档 | 内容 |
|------|------|------|
| 需求 | `001-REQUIREMENTS.md` | 13 条功能需求 + 10 条非功能需求 + 验收标准（含 3 类异常流程） |
| 设计 | `002-DESIGN.md` | 分层架构、5 个核心接口、领料流程 7 步、依赖注入组装 |
| 拆解 | `003-ISSUE_LIST.md` | 把领料流程切成可独立交付的 GitHub Issue |

这三步**顺序不能乱**：需求定义「做什么」，设计定义「怎么搭」，Issue 才能基于设计被拆出来。跳过任何一环，后面都站不稳。

### 阶段 2：定义「做对的标准」与「怎么写代码」（规范底座）

| 产出 | 文档 | 作用 |
|------|------|------|
| 测试计划 | `004-TEST_PLAN.md` | **在编码前**写好，把验收标准锁成 23 个测试用例（TC-01~23） |
| 代码规范 | `005-CODING_STANDARD.md` | 10 条核心规范（面向接口、构造注入、参数化 SQL……） |
| CI | `.github/workflows/build-test.yml` | 每次 push / PR 自动编译 + 跑全部测试 |

> 注意：测试计划写在编码**之前**。这就是 002（为什么提前写测试）讲的——先定义"什么算做对"，再去实现。

### 阶段 3：逐个 Issue 迭代开发（8 个 Issue，每个走完整闭环）

按依赖顺序一个一个做，**每个 Issue 都走「建分支 → 编码 → 本��测试 → PR → CI → 自审 → 合并」**：

| Issue | 交付物 | PR |
|-------|--------|-----|
| #1 | 解决方案 + 主项目 + 测试项目骨架 | #14 |
| #2 | 数据访问层（DbHelper + 两个 Repository） | #17 |
| #3 | 三个硬件/MES 接口定义 | #18 |
| #4 | 三个进程内 Fake 实现 | #20 |
| #7 | PickupService（领料编排 + 异常处理） | #21 |
| #8 | PickupForm（界面 + 实时进度） | #22 |
| #9 | 端到端测试（真 Service + Fake + SQLite 串联） | #23 |
| #10 | CI 跑全量测试收尾 | #24 |

每个 PR 关联对应 Issue（`Closes #N`），CI 通过 + 自审通过才合并。**这正是 005（CI）和 006（Code Review）讲的纪律落地。**

## 三、最值得讲的一段：设计中途改了

复盘里最有教学价值的，不是「计划执行得多顺」，而是**我们承认计划错了、并把它改对**。

原始设计有两处后来被推翻：

| 原方案 | 问题 | 改成 |
|--------|------|------|
| 数据库用 Access(.mdb) | 强制 x86，CI 跑不了数据库测试 | **SQLite(.db)**：AnyCPU、跨平台、CI 原生可跑 |
| 3 个独立 Mock 程序（TCP/HTTP）+ 3 个网络客户端 | 偏「系统管线」复杂度，冲淡了「工程流程」这个教学主题 | **3 个进程内 Fake**，合并成单个 Issue #4 |

这次调整把原计划的 **13 个 Issue 精简成 8 个**（原 #5/#6/#11/#12/#13 关闭）。

**为什么这件事值得专门讲：**

1. **工程是会演进的**。再周密的设计也可能在动手后暴露问题——关键不是「不犯错」，而是**早发现、敢修正**。
2. **修正要留痕**。我们没有偷偷改掉原文档假装从来如此，而是在 `000-PROJECT_STARTUP.md` 末尾追加「设计演进记录」、在 `003-ISSUE_LIST.md` 开头写明「为什么改」。**留痕本身就是工程纪律**——后人（包括三个月后的自己）能看懂当初为什么这么决定。
3. **它印证了"提前思考"的真正含义**。提前思考不是"一次想对到底"，而是"想得足够清楚，以至于发现问题时知道该改哪里、改了有什么影响"。

> 一句话：**好的流程不保证你不走弯路，但保证你走弯路时看得见、改得动、说得清。**

## 四、最终交付物清单

走完整条流程，手上有这些东西：

**文档（doc/）**
- 5 份底座文档：需求、设计、Issue 清单、测试计划、代码规范
- 1 份启动全记录（含设计演进追记）
- 1 套教学材料（teaching/，9 篇 + 索引）

**代码（src/）**
- 分层清晰：Models / Data / Hardware / Services / Forms
- 业务逻辑（PickupService）只依赖接口，**换 3 行就能从 Fake 切到真硬件**
- 一个能跑的 WinForms 程序，首次启动自建库 + 预置演示数据

**测试（test/）**
- 36 个测试：PickupService 单元测试（覆盖每条成功/失败分支）+ Repository 测试 + 无界面端到端测试
- 全部可在 CI 上原生运行

**工程设施**
- GitHub Issue / PR / CI 全程留痕
- 每次提交自动编译 + 跑全量测试

## 五、这条流程回答了开篇的问题

回到 000 问的那句话：**「为什么不能上来就写代码？」**

现在有了具体答案，对照着看：

| 000 给的理由 | 在这次开发里的体现 |
|------------|------------------|
| 建立全局思维 | 先有需求 + 设计 + Issue 拆解，才知道每段代码在整体里的位置 |
| 外化、便于审视 | 把思考写成文档，才发现 Access/独立 Mock 方案有问题并及时改掉 |
| 多角度审视 | 测试计划（怎么验）、代码规范（怎么写）、Code Review（合并前再看一遍）三道关 |
| 流程倒逼习惯 | Issue / PR / CI 强制每一步都有产出物和验收，想偷懒都难 |

而 020 拔高的那个内核——**定义问题（what）+ 选择策略（why）**——贯穿全程：

- 写需求、写测试 = 反复**定义问题**（要解决什么、什么算解决了）
- 分层、拆 Issue、改设计 = 反复**选择策略**（凭什么这么搭、这么切、这么改）

## 六、可复用的"动作清单"

抛开这个具体项目，这套流程可以套到任何新功能上：

```
1. 写需求    —— 一句话需求展开成"做什么 + 什么���做对"
2. 写设计    —— 分层、定接口、画流程；想清楚"怎么搭"
3. 拆 Issue  —— 切成能独立交付、能验收的小任务，理清依赖
4. 写测试计划 —— 编码前定义验收用例（不是写完再补）
5. 定规范+CI —— 怎么写代码、谁来自动把关
6. 迭代开发  —— 一个 Issue 一个闭环：分支→编码→测试→PR→CI→自审→合并
   ↑ 中途发现设计有问题？改它，并把"为什么改"留痕
```

**门槛在降低**：过去走这一整套成本很高，单人项目往往省掉大半，情有可原。但今天工具（尤其 AI 协作）把每一步的成本都压下来了——写文档、补测试、配 CI 都比从前快得多。**门槛低了，就更没有理由"上来就写代码"。**
