# Issue、Pull Request、CI、Code Review 详解

> 用形象的语言讲透四个核心概念 · 适合新手理解"是什么、为什么、怎么运作"

---

## 开篇：换个视角看软件开发

先忘掉代码，想象你是一个**建筑工程师**，要盖一栋楼。

你不会直接搬砖——你会先画图纸（设计）、拆成施工任务单（Issue）、每道工序单独验收（CI）、关键节点请专家会审（Code Review）、最后把合格的活合进主体（Pull Request）。

软件开发跟盖楼一样，这四个概念就是建筑行业上百年摸索出来的"怎么把活干好"的方法，只不过搬到了代码世界里。

---

## 一、Issue —— 施工任务单

### 1.1 形象理解

> **Issue 是贴在墙上的一张任务卡片。** 它说清楚三件事：**谁要干什么、干到什么程度算完、跟别的活有什么前后关系。**

想象一个建筑工地：

```
┌─────────────────────────────────────┐
│  任务单 #7                           │
│                                     │
│  [ ] 浇筑二楼承重墙                   │
│                                     │
│  图纸依据：结构施工图 S-03            │
│  验收标准：                           │
│    ☐ 混凝土标号 C30，坍落度 120±20   │
│    ☐ 垂直度偏差 ≤ 5mm               │
│    ☐ 养护 7 天后强度 ≥ 85%          │
│                                     │
│  前置任务：#5 一楼框架浇筑完成         │
│  后续任务：#9 安装二楼楼板            │
└─────────────────────────────────────┘
```

换到代码世界，Issue 就是这张卡片：

```
┌─────────────────────────────────────┐
│  Issue #7: 实现领料业务逻辑           │
│                                     │
│  要做什么：                           │
│    实现 PickupService，编排完整领料流程│
│                                     │
│  验收标准（做完的标志）：              │
│    ☐ 成功领料：查MES→找库存→取料→扫码 │
│      →上报→出料，全流程跑通            │
│    ☐ 库存不足：返回失败，不动PLC       │
│    ☐ 读码失败：放回仓位+锁定+记日志    │
│    ☐ 所有测试通过                     │
│                                     │
│  依赖：#2（数据层）#3（硬件接口）       │
│  被依赖：#8（界面层）                 │
└─────────────────────────────────────┘
```

### 1.2 本质是什么

Issue 的本质是 **"把模糊的大目标，拆成可验证的小承诺"**。

说"做一个领料系统"太模糊了——不知道从哪开始、不知道做到哪算完、不知道别人能帮你做什么。拆成 Issue 之后，每个任务都是**明确的、可验证的、可分配的小承诺**。

### 1.3 深入原理：Issue 解决什么问题

**问题一：人类大脑的"工作记忆"有限**

心理学研究表明，人一次能同时处理的信息单元大约是 4-7 个。一个软件系统有几十个模块、上百个交互——全装在脑子里不现实。Issue 把系统拆成小卡片，你一次只需要关注一张，大脑负担降到最低。

**问题二："完成了"是种模糊感觉**

没有明确的验收标准，"做完了"就是一种主观感觉——"大概能跑了"、"差不多"。Issue 用 checkbox 清单把验收标准**锁死**——标准不满足就是没做完，没得商量。

**问题三：隐藏的依赖会暗中坑你**

不写下来，依赖关系只存在于某个人脑子里。那人请假了？项目停摆。Issue 列表把所有依赖显性化，谁先谁后一目了然，任何接手的人都能看懂。

### 1.4 GitHub Issue 的工作机制

```
创建 Issue（状态: Open）
    │
    ├── 分配给人（Assignee）
    ├── 打标签（Label: bug / feature / enhancement）
    ├── 关联项目看板（Project）
    │
    ↓
有人开始干活 → 建分支、编码
    │
    ↓
提 PR（正文写 Closes #7）
    │
    ↓
PR 合并 → GitHub 自动把 Issue 状态改为 Closed
    │
    ↓
Closed 的 Issue → 变成永久档案，"当初为什么这么做"的证据
```

> **核心机制**：`Closes #N` 是一个**自动化触发器**。PR 正文里写了这句话，一旦 PR 合并，GitHub 自动关闭对应 Issue——不需要手动去关，不会忘。

## 二、Pull Request —— 变更提案

### 2.1 形象理解

> **PR 是一份"我要改主体工程"的正式申请报告。** 你带着图纸和样品，站在审核委员会面前说："这是我要改的东西，原因在这里，验证结果在这里，请过目。"

继续用建筑比喻：

```
建筑工地上，你不能自己偷偷往主体结构里加根柱子
——哪怕你的本意是好的，也可能破坏整体受力。

正确的流程是：

  1. 你在自己的工棚里（分支）把柱子做好
  2. 写好变更说明：哪里改了、为什么改、做过哪些测试
  3. 提交"变更申请"（PR）给项目总工
  4. 总工看图纸、验样品（Code Review）
  5. 自动化检测跑一遍（CI）
  6. 两个都通过 → 柱子合进主体（Merge）
```

代码世界一模一样：

```
main 分支 = 主体建筑（永远应该能住人）
feature 分支 = 你的工棚（随便折腾，不影响主体）

PR = 从工棚到主体的"门禁申请"
```

### 2.2 本质是什么

PR 的本质是 **"把零散的代码改动，打包成一个可被讨论、可被检查、可被批准或打回的对象"**。

它把"改代码"这件技术活，升维成了"提提案"这件协作活。

### 2.3 深入原理：PR 背后的 Git 数据结构

这是整个 GitHub 工作流最精妙的部分。要理解 PR，得先理解 Git 是怎么存代码的。

**Git 的核心：有向无环图（DAG）**

Git 不是"保存每个文件的每个版本"，而是**保存每次改动的"快照差异"**，然后用指针把它们串成一条链：

```
main 分支：

  A ──→ B ──→ C ──→ D （main 指针指向 D）
  ↑                
  初始               最新

你在 D 的基础上创建分支：

  A ──→ B ──→ C ──→ D （main）
                     \
                      E ──→ F （feature/issue-7，你在这干活）
```

**PR 做了什么**

当你提 PR 时，GitHub 做的事：

1. **定位分叉点**：找到 `feature` 分支从 `main` 分出去的那个 commit（上图的 D）
2. **计算差异（diff）**：把 D 到 F 之间的所有改动算出来——这就是 PR 里 "Files changed" 显示的内容
3. **检查能否合并**：判断 main 有没有在你之后又往前走了（有没有人合了别的东西？）
   - **没有** → "This branch has no conflicts with the base branch" ✅
   - **有** → 可能冲突，需要手动解决 ⚠️
4. **触发 CI**：对合并后的结果跑自动检查

```
你的分支:  D → E → F
                     \
main:    D → G → H    ↘  合并点 = (F + H) 的融合
                    
PR 页面显示的 diff = D 到 F 的差异
CI 跑的 = 合并后融合状态的代码
```

> **关键洞察**：CI 测的不是你的分支上的代码，而是**你的分支 + main 最新状态合并之后的结果**。这才能保证"你合进去之后，整个项目不会坏"。

**合并操作（Merge）的三种方式**

当你点 "Merge pull request" 时，GitHub 提供三种策略：

```
方式一：Create a merge commit（默认）
  保留你分支的完整历史，创建一个"合并提交"把你的分支和 main 拧在一起
  
  结果图：
    D ──→ G ──→ H ──→ M (merge commit, 两个爹: H 和 F)
         \         /
          E ──→ F

方式二：Squash and merge（推荐单人/小项目）
  把你分支上的多个零散 commit 压成一个干净的 commit，再放到 main 上
  
  结果图：
    D ──→ G ──→ H ──→ S (squashed commit, 只包含 F 的最终效果)
         \
          E ──→ F  (分支的这些中间提交不出现在 main 历史上)

方式三：Rebase and merge
  把你分支的提交"平移"到 main 的最新位置，保持线性历史
  
  结果图：
    D ──→ G ──→ H ──→ E' ──→ F'  (没有分叉，一条直线)
```

本项目的选择：**Create a merge commit**（保留完整痕迹）。

### 2.4 PR 作为"容器"

PR 本身不干活——它是一个**容器**，装了三个东西：

```
        ┌─────────── 一个 PR ───────────┐
        │                                │
  PR 描述（做了什么、为什么、怎么验证）      │
        │                                │
  diff（代码改了什么，行级对比）            │
        │                                │
  审查线程（CI 报告 + 人的评论 + 讨论记录） │
        │                                │
        └────────────────────────────────┘
```

没有 PR 这个容器，CI 不知道"什么时候该跑"、Review 不知道"在哪里看"。PR 是整个流程的**结构骨架**。

## 三、CI（持续集成）—— 自动质检流水线

### 3.1 形象理解

> **CI 是一条工厂流水线末端的自动质检机。** 每个零件（代码改动）放上传送带，机器自动量尺寸、测强度、查瑕疵——合格的放行，不合格的当场打回去。

你可以想象一座汽车工厂：

```
以前（没有 CI）：
  工人在自己工位上装好零件 → 靠自觉检查 → 装上车
  问题：有人偷懒没查、有人忘了、有人"我感觉没问题"
  结果：瑕疵零件混进整车 → 出厂后才发现 → 召回成本巨大

现在（有 CI）：
  每个零件装好 → 放上传送带 → 自动检测站
  
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ 尺寸测量  │ → │ 强度测试  │ → │ 瑕疵扫描  │ → ✅ 放行
  └──────────┘    └──────────┘    └──────────┘    ✗ 打回
```

换到代码世界：

```
以前（没有 CI）：
  你改完代码 → 本地跑了一下 → 感觉没问题 → 合进 main
  问题：你本地装了某个库别人没装、你忘了跑全部测试、
        你的改动弄坏了别人的模块但你不知道……
  结果：bug 混进 main → 别人拉下来发现跑不起来 → 开始排查 → 浪费时间

现在（有 CI）：
  你提 PR → 自动触发 CI → 在云端干净机器上：
  
  ┌────────┐    ┌────────┐    ┌────────┐
  │ 拉代码  │ → │ 编译    │ → │ 跑测试  │ → ✅ All checks passed
  └────────┘    └────────┘    └────────┘    ✗ Build failed / Tests failed
                   ↑                        ↑
              拦住低级错误              拦住逻辑回归
```

### 3.2 本质是什么

**CI 是把"质量检查"这件事，从靠人自觉变成了靠系统强制。**

它回答一个最根本的问题：**"测试写好了，谁来保证它每次都被运行？"** 答案是：不靠人记住，靠流程自动触发。

### 3.3 深入原理：CI 为什么能用"干净环境"解决问题

**经典的"在我机器上能跑"困境**

每个开发者的本地环境都不一样：

- 你装了 .NET 9，同事装的 .NET 8
- 你 Windows，同事 Mac
- 你装了某个全局工具，同事没装
- 你的环境变量配了某个路径，同事没配

这就导致一个常见悲剧：代码在你机器上好好的，合进 main 之后同事拉下来跑崩了。

**CI 怎么破局**

CI 每次都**从零开始**——

```
你提 PR
  ↓
GitHub Actions 在云端申请一台全新的虚拟机
  ↓
这台机器：
  - 只装操作系统（Windows / Ubuntu / macOS，干净的）
  - 没有任何你的"私货"（没有全局工具、没有特殊配置）
  - 按你指定的步骤从零搭建环境：
    ① checkout 代码
    ② 装指定版本的 .NET SDK
    ③ 还原依赖包
    ④ 编译
    ⑤ 跑测试
  ↓
如果这台"裸机"上能跑通 → 说明你的代码不依赖任何特殊环境
如果跑不通 → 说明你的代码有问题（依赖了某个你没意识到的本地私货）
```

> **"干净环境"的本质是去除了所有隐性依赖**——让你的代码在"最苛刻"的条件下暴露问题。能在裸机上跑通的代码，在任何人的机器上都能跑通。

**幂等性：每次结果都一样**

CI 的另一个关键特性：**同样的代码，跑一万次，结果都一样**。

这依赖于 CI 环境的**确定性**：

- 同样的操作系统版本
- 同样的 SDK 版本
- 同样的依赖包版本（由 `package.lock` / `nuget.lock` 锁定）
- 没有任何"上次运行留下的残留"

如果有残留（比如上次运行生成的临时文件没删、数据库没清），就会出现"时过时不过"的幽灵 bug——这是开发者最怕的东西。CI 每次从零搭建消除了这种不确定性。

### 3.4 CI 配置文件解剖

本项目的真实 CI 配置（`.github/workflows/build-test.yml`），逐层拆解：

```yaml
# 第一层：给这个 workflow 起个名字
name: Build and Test

# 第二层：什么时候触发
on:
  push:
    branches: [ main ]        # push 到 main 就触发
  pull_request:
    branches: [ main ]        # 向 main 提 PR 就触发

# 第三层：做什么事情
jobs:
  build-test:                 # 作业名（随便起）
    runs-on: windows-latest   # 在什么操作系统上跑（WinForms 必须是 Windows）

    steps:                    # 一系列步骤，按顺序执行
      - name: Checkout        # 步骤 1：拉代码
        uses: actions/checkout@v4

      - name: Setup .NET      # 步骤 2：装 .NET 9
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '9.0.x'

      - name: Restore         # 步骤 3：下载依赖包
        run: dotnet restore

      - name: Build           # 步骤 4：编译
        run: dotnet build --configuration Release --no-restore

      - name: Test            # 步骤 5：跑测试
        run: dotnet test --configuration Release --no-build
```

这份 46 行的文件 = **一套永远在线的自动质检系统**。没有服务器、没有维护、免费使用（公开仓库）。

### 3.5 CI 和 Git 的本质区别

```
Git（版本管理）：               CI（持续集成）：
┌──────────────────┐          ┌──────────────────┐
│ 记录你做了什么      │          │ 检查你做得对不对    │
│ 被动存档           │          │ 主动验证           │
│ "改了什么？谁改的？" │          │ "项目还正常吗？"    │
│ 历史账本           │          │ 质检员             │
└──────────────────┘          └──────────────────┘
```

**Git 不会告诉你代码是对是错**——它忠实地记录了你写了一个 bug，就像账本不会告诉你哪笔账算错了。CI 补上了这个空缺：每次改动后自动判断"项目还能正常工作吗"。

## 四、Code Review —— 人工专家会审

### 4.1 形象理解

> **Code Review 是把你的代码图纸拿到评审会上，让专家们用挑剔的眼光过一遍。不是不信任你——而是任何人都会有盲区，多一双眼睛就少一个隐患。**

建筑行业有个机制叫 **"图纸会审"**：

```
施工单位画好施工图 → 不是直接拿去盖楼
    ↓
先组织会审会议：
  - 结构工程师：承重计算对吗？
  - 给排水工程师：管道和电缆打架吗？
  - 消防工程师：逃生通道合规吗？
  - 造价工程师：有没有浪费材料的地方？
    ↓
每个专业的人从自己的角度审视同一张图
    ↓
问题改在图纸上（成本低）←→ 别等问题出在工地上（成本高）
```

代码审查是完全一样的逻辑：

```
你写好代码 → 不是直接合并
    ↓
审查者从不同角度审视同一份 diff：
  - 设计角度：分层合理吗？职责清晰吗？
  - 逻辑角度：异常处理全吗？边界情况考虑了吗？
  - 可维护性：三个月后别人能看懂吗？
  - 安全性：有没有注入风险？
    ↓
问题改在 PR 上（成本低）←→ 别等问题出在生产环境（成本高）
```

### 4.2 本质是什么

**Code Review 是 CI 的大括号之外、机器永远做不到的那部分质量保障。**

CI 能保证代码**没有明显的错**（能编译、测试过），但保证不了代码**是好的**（设计优雅、逻辑清晰、可维护）。后者需要人类的判断力。

### 4.3 深入原理：为什么人眼审查不可替代

**原理一：正确 ≠ 好**

一段代码可以完美通过所有测试，但仍然是烂代码：

```csharp
// "正确"但糟糕的代码：把所有逻辑塞在一个 500 行的按钮点击事件里
// CI 会说：✓ 编译通过、测试通过。没问题。
// Review 会说：✗ 不可维护、不可测、不可复用。重写。

private void ButtonStart_Click(object sender, EventArgs e)
{
    // 500 行：查数据库、调硬件、做判断、更新界面……全在这
}
```

CI 只判断"对不对"（correctness），不判断"好不好"（quality）。后者需要人的经验和品味。

**原理二：开发者的"确认偏误"**

心理学上的确认偏误（Confirmation Bias）：**人会不自觉地寻找支持自己判断的证据，而忽略反对的证据。**

写代码时，你已经进入"怎么做"的模式——满脑子是这个函数怎么实现。这时候看自己的代码，眼睛会自动跳过"有问题"的地方，因为大脑认为"我已经想过了，大概没问题"。

Code Review 的审查者没有这个偏见——他是第一次看这段代码，用"这写的什么东西"的眼光审视。很多你下意识跳过的问题，在他眼里是显眼的红灯。

**这就是为什么"隔夜再回来看"有效**：隔了一夜，你大脑里"怎么实现"的记忆淡了，你才能用接近审查者的眼光重新看自己的代码。

**原理三：知识在团队中的扩散**

Code Review 有个隐藏收益，比"挑错"更大——

> **审查者在审代码的过程中，学会了这部分代码是怎么工作的。**

这意味着：

- 原本只有作者懂的模块，至少有两个人懂了
- 原作者请假/离职，系统不会出现"没人敢动"的模块
- 审查者可能会发现："这个功能隔壁模块已经有了，别重复造轮子"（跨模块知识）

这是 Code Review 超越"质量控制"的更深层价值——**知识共享**。

### 4.4 CI 和 Code Review 的分工

```
                    你的代码
                       ↓
        ┌──────────────┴──────────────┐
        ↓                              ↓
      CI                              Code Review
 （机器，自动）                      （人类，人工）
        ↓                              ↓
  查什么：                          查什么：
  ┌──────────┐                    ┌──────────────┐
  │ 能编译吗  │                    │ 设计合理吗     │
  │ 测试全过吗 │                    │ 职责清晰吗     │
  │ 格式合规吗 │                    │ 可维护吗       │
  │ 无编译警告 │                    │ 注释说清了吗   │
  └──────────┘                    │ 偏离 Issue 了吗│
        ↓                          └──────────────┘
  守底线                                提上限
  （不达标的，进不来）                   （能跑的，让它更好）
        ↓                                ↓
        └──────────────┬──────────────┘
                       ↓
              两个都通过 → 才能合并
```

> **一句话分工**：CI 拦住"坏的"（prevent bad），Code Review 追求"好的"（pursue good）。缺一个，要么代码能跑但烂，要么代码永远合不进去。

### 4.5 单人项目为什么还要做

**天然短板**：再怎么"隔夜再看"、切换角色，终究是**同一个脑子**——你不知道的问题，自己审一百遍也发现不了。

**破局方式——AI 当"第二双眼睛"**：

把 diff 交给 AI，让它用你不具备的角度审视：

```
你的视角：我写得挺顺的，逻辑都对
AI 的视角：
  "第 42 行，这里如果 input 为 null 会怎样？"
  "第 78 行调用了异步方法但没有 await，是有意的吗？"
  "这个异常分支在测试里没有对应的用例"
```

AI 不会替代你的判断，但它能**弥补"同一个脑子"的视角盲区**——它见过的代码模式比你多，能指出你没意识到的潜在问题。

## 四者关系总览

```
                              Issue #7
                     "实现领料业务逻辑"
                     ┌──────────────────┐
                     │ 做什么？          │
                     │ 验收标准？        │
                     │ 依赖谁？被谁依赖？ │
                     └──────┬───────────┘
                            │ 对应
                            ↓
                     feature/issue-7-pickup-service
                     ┌──────────────────────────────┐
                     │  你的分支（工棚）              │
                     │  写代码、本地测试               │
                     └──────────────┬───────────────┘
                                    │ 打包提案
                                    ↓
                         ┌─── Pull Request ───┐
                         │                     │
                         │  做了什么？为什么？   │
                         │  怎么验证的？        │
                         │  Closes #7          │
                         │                     │
                         │  ┌── CI ──────────┐ │
                         │  │ 编译 ✓          │ │
                         │  │ 测试 ✓ 全绿     │ │  ← 自动触发
                         │  │ 格式 ✓          │ │
                         │  └────────────────┘ │
                         │                     │
                         │  ┌ Code Review ───┐ │
                         │  │ 设计合理吗？     │ │
                         │  │ 逻辑完整吗？     │ │  ← 人工审查
                         │  │ 三个月后能懂吗？ │ │
                         │  └────────────────┘ │
                         │                     │
                         │  都通过 → Merge ────→ main
                         │  Closes #7 → Issue 自动关闭
                         └─────────────────────┘
```

### 一句话记住四个概念

| 概念                   | 一句话                                   | 建筑比喻         |
| ---------------------- | ---------------------------------------- | ---------------- |
| **Issue**        | 施工任务单——做什么、什么算做完         | 贴墙上的任务卡   |
| **Pull Request** | 变更提案——把改动打包成一个可审查的对象 | 变更申请报告     |
| **CI**           | 自动质检机——每次改动自动编译+测试      | 传送带上的检测站 |
| **Code Review**  | 专家会审——人工判断设计是否合理         | 图纸会审会议     |

### 它们的共同内核

Issue → PR → CI → Review → Merge，这条链每一步都在做同一件事：

> **把"靠人自觉"的事，变成"靠流程保障"的事。**

| 靠人自觉（脆弱）       | 靠流程保障（稳定）    |
| ---------------------- | --------------------- |
| "我记得要写测试"       | CI 自动跑，不写就红   |
| "我知道代码大概没问题" | Review 人眼过一遍     |
| "我回头去关那个 Issue" | Closes#N 自动关       |
| "我本地能跑应该就行"   | CI 干净环境重新搭一遍 |

这不只是工具使用——这是**工程文化**的体现：不靠某个人的英雄主义，而靠流程和系统，让普通人也能稳定产出可靠的成果。
