## 二、文档之间的关系

整个教学体系是一条**精心设计的讲课主线**：

```
000 立总纲：动手前先思考，为什么？
  ↓
001 拆 Issue      ─┐
002 提前写测试     ├── 具体动作的 why（倒逼全局思考 + 定义"做对的标准"）
003 测试工具       ─┘   配套工具教学（xUnit / NSubstitute / 两种 Fake 分工）
  ↓
004 代码规范      ─┐
005 Pull Request   ├── 流程与质量三连（制定→提交→审查→合并）
006 CI             │   PR 是"容器"，CI 和 Review 都装在里面
007 Code Review   ─┘
  ↓
010 自我剖析      —— 元说明，解释"为什么这么讲"（选讲，可穿插开场）
  ↓
020 收尾拔高      —— 所有动作收束为"定义问题 + 选择策略"，首尾呼应 000
030 过程复盘      —— 真实时间线复盘，重点讲设计修正与留痕
040 演示指南      —— 不讲理念，只讲操作
```

### 关键关系

1. **000 ↔ 020 首尾呼应**：000 问"为什么不能上来就写代码"，020 答"因为动手前在做两件事——定义问题、选择策略"
2. **001/002/003 是一组**：都讲"编码之前该做什么"，从 Issue 规划 → 测试计划 → 测试工具，逐层深入
3. **004 → 005 → 006 → 007 是链条**：规范制定（004）→ PR 提交（005）→ CI 自动检查（006）→ Code Review 人工把关（007），一步步走完合并前的完整关卡
4. **030 是 000-007 的真实佐证**：所有理论在真实项目里"跑了一遍"，用 commit/PR 留痕说话
5. **010 是"元认知"层**：跳出内容本身，解释整套教学的设计逻辑

### 共同基调

- **从 why 出发**，每条理由配反例/对比
- **落到真实项目**（capillary-exercise）做锚点，不空谈
- **结尾点"门槛"**：过去走流程成本高、情有可原；今天工具（尤其 AI）降低了门槛，更有理由用好方法

## 三、是不是在教授 GitHub 如何使用？

**不是。这套材料不是在教 GitHub 怎么用。**

它真正在教的是**软件工程的思维方式和开发习惯**，具体来说：

- **教的不是"怎么用 GitHub"**，而是"为什么要用 Issue/PR/CI/Review 这套流程做软件"
- **教的不是工具操作**（how），而是**工程判断力**（why）——这正是 010（自我剖析）反复强调的核心定位
- 005（PR）、006（CI）、007（Review）虽然涉及 GitHub，但它们的落点不是"gh pr create 命令怎么敲"，而是"PR 作为提案容器的意义""CI 本质是工程纪律的跃迁""Review 是机器与人的分工"

README 里明确写了定位：

> 本目录收录"流程驱动开发"培训的理念讲解材料

010（自我剖析）讲得更直白：

> 我是顾问，不是培训师。顾问交付的是判断力，不是可替代的技能。how 你们能自学，why 才是我该留给你们的。

### 正确理解

这套材料是：**以 capillary-exercise 项目为案例，借助 GitHub 生态（Issue/PR/CI）作为载体，教授"流程驱动开发"的工程思维**。GitHub 只是"用到的工具"，不是"教学的目标"。

## 四、这个项目本身是干嘛的

项目有**两层身份**：

### 第一层：真实的软件产品

**劈刀（Capillary）自动存取管理系统**——工厂产线上的 WinForms 桌面程序，管理劈刀领料流程：

- 操作员输入工单号 + 机台号
- 系统自动查 MES 确定劈刀类型 → FIFO 找库存 → 控制 PLC 取料 → 扫码验证 → 上报出料
- 本期只做**领料（Pickup）**，存料、退料不在范围内

技术栈：C# .NET 9 WinForms + SQLite + xUnit/NSubstitute。用**进程内 Fake** 模拟硬件，无需真实硬件即可运行和演示。

### 第二层：教学案例

这个项目的真正目的是**拿它当教学锚点，完整演示一遍软件工程闭环**：

```
需求 → 设计 → Issue 拆解 → 测试计划 → 代码规范/CI → 迭代开发
```

核心教学主张：**动手写代码之前，先把"要做什么、怎么搭、怎么验证"想清楚，并把思考过程沉淀成可追溯的资料**（需求文档、设计文档、Issue、测试计划）。

> 这是一个"用真实代码跑通一套工程流程"的练习项目——代码本身能用，但更重要的产出是走完流程后的那套思维习惯。

## 五、核心教学内容："流程驱动开发"详解

整套 teaching 材料的核心教学目标，就是**流程驱动开发（Process-Driven Development）**。下面从定义、为什么、怎么做、底层机制四个层面展开。

### 5.1 什么是"流程驱动开发"

**一句话定义**：

> 在写第一行代码之前，先走完一套结构化的思考与规划流程——需求、设计、任务拆解、测试计划、规范制定——把"要做什么、怎么做、怎么验证"全部想清楚并沉淀为文档，然后再进入编码阶段。

它不是瀑布模型那种"一次写完所有文档再开发"，而是**每个功能/每个 Issue 都先思考再动手**。流程的作用是倒逼你思考，文档的作用是留下思考的痕迹。

### 5.2 为什么要走流程（四个根本理由）

#### 理由一：倒逼你建立全局观

"上手就写"最大的问题是**视野被锁死在当前这一行代码**——注意力全在"这个函数怎么实现"，忘了模块怎么衔接、数据怎么流转。

走流程的本质是**逼你在写代码之前，先把整个系统在脑子里过一遍**：

- 写设计文档 → 逼你回答"系统由哪些部分组成、怎么分层"
- 拆 Issue → 逼你回答"每个部分的边界在哪、谁依赖谁、谁先谁后"

> 你没法在不理解全局的情况下，写出合理的设计、拆出合理的 Issue。久而久之，这种"被逼着想全局"就内化成全局思维——先看森林、再看树木。

#### 理由二：把思考外化成资料，支撑持续迭代

人脑里的想法是模糊的、跳跃的、容易自我欺骗的。很多漏洞只有在**把想法写下来**的那一刻才暴露——写设计时发现接口对不上、拆 Issue 时发现功能无法独立测试、写测试计划时发现边界情况从没考虑过。

更深层的价值在**长远**：

> **代码只记录了"怎么做（how）"，没记录"为什么这么做（why）"。** 半年后要改，只有代码的人得反推设计意图——而且未必猜得对。有了需求、设计、Issue 这些"图纸"，迭代才有依据，知道哪能改、为什么这样设计。

#### 理由三：站在前人经验上，让流程值得信赖

流程驱动开发不是谁拍脑袋发明的，而是**几十年软件工程实践沉淀的经验**——无数项目用教训换来的。更重要的是，这些经验已经**固化进了工具**：

- Git/GitHub → 版本管理、可追溯
- Issue → 任务拆解、进度跟踪
- Pull Request → 代码审查的必经环节
- CI → 每次改动都自动测试

> 你顺着工具走，就不知不觉走在了被验证过的正确路径上。

#### 理由四：多角度审视同一个问题

流程的每个环节，都是对"同一个系统"的一次**独立检视，但角度不同**：

| 环节        | 视角        | 问的问题                           |
| ----------- | ----------- | ---------------------------------- |
| 需求        | 用户 / 产品 | 要解决什么问题？什么算解决了？     |
| 设计        | 架构师      | 怎么分层？接口怎么定？数据怎么流？ |
| Issue       | 项目经理    | 怎么拆成可交付的任务？依赖是什么？ |
| 测试计划    | 测试 / 质保 | 怎么验证？边界在哪？异常怎么处理？ |
| Code Review | 审查者      | 设计合理吗？有没有更简单的做法？   |

**多人协作**时，这些视角天然分给不同的人；**单人项目**时，流程逼你一个人轮流扮演多个角色，弥补"只有一个脑袋、一个视角"的天然缺陷。

### 5.3 怎么做：流程的六个步骤

教学材料 030（开发过程总结）提供了完整的**可复用动作清单**：

```
① 写需求   → 把一句话需求展开成"做什么 + 什么算做对"
② 写设计   → 分层、定接口、画流程；想清楚"怎么搭"
③ 拆 Issue → 切成能独立交付、能验收的小任务，理清依赖
④ 写测试计划 → 编码前定义验收用例（不是写完再补）
⑤ 定规范+CI → 怎么写代码、谁来自动把关
⑥ 迭代开发 → 一个 Issue 一个闭环
```

#### 步骤详解

**① 需求**：将模糊的一两句话展开为结构化的功能需求（FR-01~13）、非功能需求、验收标准。关键不是写得多全，而是**逼你把模糊目标变成可验证的描述**。

**② 设计**：定义分层架构（Models / Data / Hardware / Services / Forms）、核心接口（IPlcController / IScanner / IMesService）、业务流程（领料 7 步）。关键输出是**接口定义**——接口是各层之间的"合同"。

**③ Issue 拆解**：基于设计，把工作切成可独立交付、可独立验收的任务。关键判断标准不是"一个 Issue = 一个类"，而是**"这个 Issue 做完，能独立验证一个可交付的价值"**。

**④ 测试计划**：在编码前写好验收标准和测试用例。关键转变——测试从"验证工具"变成**"思考工具"**：想不清楚怎么测，说明还没想清楚要做什么。

**⑤ 规范 + CI**：定编码规范（10 条核心规则），配 CI（每次提交自动编译 + 测试）。规范分两类——格式类（机器自动强制）和设计类（人 Code Review 把关）。

**⑥ 迭代开发**：每个 Issue 走完整闭环：

```
建分支 → 编码 → 本地测试 → 提 PR → CI 自动验证 → Code Review → 合并
```

一个 PR 对应一个 Issue（`Closes #N`），CI 绿了 + 审查通过才合并。

### 5.4 底层机制：流程如何倒逼习惯

整套教学反复出现一个词：**"流程倒逼习惯"**。这是流程驱动开发最底层的运作逻辑：

> 你不能指望靠"意志力"和"自觉"来养成好习惯——人会忘、会偷懒、会自我妥协。而是用**流程设计**，让"正确的做法"成为"唯一的做法"。

具体怎么倒逼：

| 你想跳过的事             | 流程怎么拦住你                                         |
| ------------------------ | ------------------------------------------------------ |
| 不想写设计，直接写代码   | 不写设计就拆不出合理的 Issue，拆不出来就不知道先做哪个 |
| 不想写测试               | 每个 Issue 有验收标准，CI 自动跑测试，不写就不绿       |
| 改完直接合，不审查       | PR 机制：不通过 CI 和 Review 就合不进去                |
| 测试坏了假装没看见       | CI 每次自动跑，红了挡合并，逼你当场修                  |
| 设计有更好的想法但懒得改 | 文档留痕的纪律：改了就要写清楚"为什么改"               |

> **好习惯不靠意志力硬扛，靠流程的"不得不"。**

### 5.5 最终的内核：两个词

教学材料 020（收尾）把所有"提前思考"的动作收束为两个核心概念：

| 内核                       | 含义                                   | 在流程中的体现                                        |
| -------------------------- | -------------------------------------- | ----------------------------------------------------- |
| **定义问题（what）** | 想清楚到底要解决什么问题、什么算解决了 | 写需求、写"做对的定义"（测试）、明确每个 Issue 的边界 |
| **选择策略（why）**  | 凭什么这样分层、这样拆、这样排顺序     | 设计决策、Issue 粒度与依赖、技术选型权衡              |

**流程中的所有动作——写需求、画设计、拆 Issue、写测试——本质上都是在把"提前思考"这件潜意识的事，用看得见的动作显性化出来，让它可以被刻意练习。** 流程是显性化的脚手架，定义问题和选择策略是脚手架要托起的那个内核。

### 5.6 今天为什么更应该走流程

整套材料每篇结尾都点"门槛降低"这个主题：

| 过去                   | 现在                                   |
| ---------------------- | -------------------------------------- |
| 写设计文档费时费力     | AI 可帮起草、帮整理思路                |
| 拆 Issue 需要大量经验  | AI 可根据设计给出拆解建议              |
| 提前写测试又难又费时   | AI 可列出边界情况、生成测试骨架        |
| 搭 CI 要自建服务器     | GitHub Actions 一个 YAML 文件搞定      |
| 单人做不了 Code Review | AI 当"第二双眼睛"审 diff               |
| 规范靠人背、靠人查     | AI 生成代码时主动遵守，CI 自动检查格式 |

> 过去：走流程成本高 → 跳过情有可原
> 现在：工具（尤其是 AI）大幅降低成本 → **有理由用更好的方法**

这不是道德要求，而是理性判断：当好方法的成本降下来了，继续用差方法就不划算了。

## 六、涉及 GitHub 的操作

整个流程驱动开发在本项目中依赖的 GitHub 操作共 **5 个**，按开发顺序排列：

| 操作 | 做什么 | 在流程中的位置 |
|------|--------|--------------|
| **Issue** | 在 GitHub 上创建任务卡片，描述做什么、验收标准是什么。一个 Issue = 一个可独立交付的工作单元 | 阶段 ③ 拆 Issue，开工前建好 |
| **Branch** | 每个 Issue 建一条独立分支（如 `feature/issue-7-pickup-service`），在上面编码，不直接动 `main` | 阶段 ⑥ 迭代开发的第一步 |
| **Pull Request** | 分支代码写完后，发起合并请求。写清"做了什么、为什么、怎么验证"，正文关联 `Closes #N`（合并后自动关闭对应 Issue）。PR 是 CI 和 Review 的**载体** | 编码完成 → 提 PR |
| **CI（GitHub Actions）** | PR 自动触发云端编译 + 跑全部测试。全绿才允许合并，红了必须修。配置文件 `.github/workflows/build-test.yml` | PR 提交后自动运行 |
| **Code Review** | 在 PR 页面上审 diff——设计是否合理、逻辑是否清晰、有没有偏离 Issue 目标。CI 管"能不能跑"，Review 管"好不好" | CI 绿了之后、合并之前 |

### 完整的 GitHub 操作流

```
① 建 Issue（GitHub 网页 / gh issue create）
       ↓
② 建分支（git checkout -b feature/issue-N-xxx）
       ↓
③ 编码 + 本地测试（dotnet test）
       ↓
④ 提交推送（git commit + git push）
       ↓
⑤ 提 PR（gh pr create，正文写 Closes #N）
       ↓  ← GitHub Actions 自动触发
⑥ CI 自动编译 + 跑测试
       ↓  全绿
⑦ Code Review（在 PR 页面上审 diff）
       ↓  通过
⑧ 合并 PR → Issue 自动关闭
```

### 关键要点

- **PR 是核心枢纽**：Issue、CI、Review 全部围绕 PR 运转——Issue 是"做什么"、PR 是"怎么做的提案"、CI 和 Review 是"把关"
- **CI 是自动关卡**：配一次 `.github/workflows/build-test.yml`（几十行 YAML），之后每次提 PR 自动生效，无需人工触发
- **`Closes #N` 是桥梁**：写在 PR 正文里，PR 一合并，GitHub 自动关闭对应 Issue，改动和需求挂上钩
- **不需要 push 到 main**：所有改动都在分支上，合并通过 GitHub PR 页面操作，`main` 始终保持可发布状态

> 这 5 个操作是 GitHub 生态为流程驱动开发提供的**基础设施**。教学的重点不在于这些操作怎么点、命令怎么敲（那是 how），而在于每个操作背后承担的**工程角色**（那是 why）。
