一、核心流程：

Issue把模糊的大目标或者功能拆分成具体的小功能，指定谁来完成该功能，类似于开发文档；lssue的内容设计需要长时间的讨论，整个项目的框架要想清楚，制定好计划再开发。

分支是指向某个版本的指针，目的是当你想要增加功能时，需要创建新分支来完善新功能，main指向不变，这样不会影响应用的使用，当新功能完成后，再将main指向新版本，在上线，主要不影响开发

分支上的改动版本想要与main主体合并，需要提出Pull Request ，修改项目功能（不会影响项目的原版本的主体），这是一种协作方式，别人写的修改版想要与主体合并，它是一个表单，写明做了什么，为什么要怎么做，测试方法等信息。

当你提出PR请求来更新项目版本，提交时会触发CI检查（会在GitHub云端跑一遍修改后的代码），没有问题就将该分支上的功能版本合并到main上即项目主体，有问题会被打回，不容许合并。

Code Review人工会审，前面只检查能不能正常运行，该阶段检查的是代码逻辑怎么样，设计结构是否清晰，是否好维护等问题的检查，觉得没有问题了就可以合并了。

提前写测试，是为了让自己对问题理解的更深刻，输入输出是什么，异常情况要怎么处理，边界在哪里；完成后再写测试会陷入如何实现的如何测试，不能真正的发现程序的问题。
